# Handoff — 2026-05-17 [00:43 session start]

## Session Goal
Make the per-bucket Claude session restore feature (24f2300, shipped untested) actually work end-to-end. User reported the resume wasn't firing on relaunch; goal was to debug live, fix every gate that turned out to be broken, and reshape the workflow so the user controls when restore happens.

## What Got Done
Commit `e042f48` — "feat: harden per-bucket Claude session restore + add manual load action":
- **Close-handler robustness** (`src/components/TerminalsView.tsx`): wrap `persistProjectTerminals` in a 1.5s `Promise.race` timeout so `win.destroy()` always runs even if Rust hangs. Added `[close] persist starting/resolved/timeout/destroying` console.info lines for debugging.
- **App-level ⌘Q orchestration** (`src-tauri/src/main.rs`): switched from `.run(generate_context!())` to `.build()?.run(callback)`. On `RunEvent::ExitRequested`, close both `main` and `project-*` windows (NOT just `project-*` — main usually hosts `lastProject`, and its persist was silently dropped before). Polls until all windows destroyed (3s max), then `app.exit(0)`. Uses `label.as_str() == "main"` to dodge the `&String == &str` Rust comparison error.
- **⌘⇧W Close Window menu item** (`src-tauri/src/main.rs`): new MenuItem with id `window_close`, accelerator `CmdOrCtrl+Shift+W`, routed to focused window's `close()` inline alongside `window_next`/`window_prev` in `on_menu_event`. `⌘W` stays bound to Close Terminal (which is the cause of the "red traffic-light doesn't close" UX issue — the menu binding swallowed window-close).
- **Session-mtime cutoff 6h → 48h** (`src-tauri/src/session_restore.rs:8`): the original 6h was too tight for overnight gaps; user's `.jsonl` files were getting silently filtered out at persist time.
- **eprintln diagnostics** (`src-tauri/src/commands.rs::persist_project_terminals`): `[persist] enter ...`, `[persist] gate=true`, `[persist] cwd=X detected_uuid=...`, `[persist] wrote N rows`. Console.app filter on `[persist]` shows the full chain.
- **Per-bucket "Load active Claude sessions" action**: new Rust command `load_active_claude_sessions_for_bucket(bucket_id) -> usize` (commands.rs) + new store query `list_persisted_project_ids_in_bucket` (store.rs) + new IPC wrapper `loadActiveClaudeSessionsForBucket` (ipc.ts) + new context-menu button in `BucketsList.tsx` (disabled when bucket empty, MENU_H bumped 120 → 160 to fit).
- **Removed launch-time auto-load** (`src/App.tsx`): main window no longer reads `lastProject` or auto-spawns persisted project windows on mount. Launcher starts empty; user picks a bucket and triggers restore manually. `lastProject` import dropped.

DB state verified mid-session via `sqlite3` against `~/Library/Application Support/com.mahmutsalman.lexical-emerson/state.db` — confirmed 2 rows survived from the working close-and-reopen tests (lexical-emerson `07615da4-…`, SpringProjects1 `630fc6c0-…`).

## What Was Tried and Abandoned
- **Approach**: Auto-spawn every persisted project window on main-mount (initial design — committed and then reverted within the same session).
  - **Why we walked back**: User pointed out the obvious — "what if there are buckets with claude code opened last time?" Multiple buckets with persisted sessions would create a window-storm at launch. Also suspected (but not confirmed) source of the "loads twice then stops" perception bug because of the race with `setCurrentProject(lastProject)` setting `main_project_id` after the spawn loop had already started.
  - **Lesson**: Treat persist-restore like restore from a snapshot, not auto-replay of the last desktop state. Surface it as a discrete action the user invokes per bucket.
- **Approach**: First ⌘Q handler iteration only closed `project-*` windows, not `main`.
  - **Why we walked back**: User reported "I close the app by clicking on quit and when I open it, only the Lexical Emerson opens." Once we mapped the persist chain, the missing piece was main: when main hosts `lastProject`, ITS `TerminalsView.onCloseRequested` is the only path to persist that project — and skipping main silently dropped it. Updated filter to `label == "main" || label.starts_with("project-")` in both the close loop and the polling loop.
  - **Lesson**: When orchestrating window-close ordering at exit time, NEVER assume "project windows" means `project-*` labels exclusively. Main is part of the project surface in this app.
- **Approach**: ⌘W (Cmd+W) as the natural "close window" keybinding.
  - **Why we walked back**: It was already bound to `terminal_close` in the menu, which closes a terminal tab and immediately auto-spawns a fresh one (per `TerminalsView`'s "ensure ≥1 tab" effect), so the window stays open and persist never fires. Resorting to the red traffic-light button revealed the next bug (the 1.5s-less await on persist could hang the destroy). Added ⌘⇧W as the proper keyboard path matching Chrome/VS Code/iTerm2.
  - **Lesson**: Menu accelerators conflict silently with OS defaults. The shadowed default doesn't fall through.

## Decisions Made
- **Per-bucket user-driven restore over launch-time auto-restore** — no ADR (small enough scope, no architectural ripples). Captured here + in STATUS recent-decisions. Likely won't bite anyone enough to warrant a full ADR unless a future refactor revives auto-restore as an option.

## Open Questions for Next Session

# Handoff — 2026-05-17 [09:30 session start]

## Session Goal
User reported: "in bucket 3d view, we show project notes, but it is not clickable and i cant click on a note and open it." Starting from that single bug, the session expanded into a full UX overhaul of the 3D bucket workspace — notes opening, scroll-wheel routing, project-name visibility, debug surface, focus model, WebGL stability, and cross-window event scoping. Each new symptom drove the next plan-mode round.

## What Got Done
Two commits, both verified by the user.

Commit `7ea36ad` — "feat: 3D bucket workspace — open notes by click, Enter-to-focus, WebGL fallback":
- **Click-to-open notes** (`ProjectNotesPanel.tsx`, `BucketWorkspace.tsx`, `NotesModal.tsx`): `<li>` → `<button>` so clicks land in CSS-3D-rotated panes. Single-click `.pnr-item` sets selection AND calls `onOpenEditor(n.id)`. A synchronous `lexical:notes-open-hint` window CustomEvent ships the noteId to `NotesModal`, which stores it in `pendingOpenNoteId` and prefers it over the default-first-note when the Tauri `menu://notes-open` event fires next. List-only race-safety guard (`list.some(...)`) in case the hinted note was deleted between hint and open.
- **Wheel scroll on facing terminal** (`BucketWorkspace.tsx` onMount): non-passive `wheel` listener on `.bw-stack` mirrors `TerminalsView.tsx:430–445`. xterm native wheel can't reach panes behind the CSS-3D perspective ancestor; we intercept and drive `handle.scrollLines()` on the facing terminal. Magnitude conversion: `Math.max(1, Math.round(|deltaY|/18))`.
- **Always-visible project chip** (`BucketWorkspace.tsx` + `bucket-workspace.css`): pill at top-left in 3D mode (originally top-right; moved after collision with Re-scan/Debug buttons). Tinted with active project's PALETTE accent on the left border. Fades to opacity 0 via `~` sibling selector when `.bw-header-trigger` or `.bw-header` is hovered. Required reordering the chip JSX to come AFTER `</header>` in DOM. Reactivity: a `createMemo<chipAccent>` keyed off `activeProject()` plus inline `proj()` accessor reads — the earlier `const p = proj()` snapshot stayed stale across ring switches because non-keyed `<Show>` only re-runs the render prop on falsy→truthy.
- **Toggleable event-debug overlay** (new `BucketWorkspaceDebugLog.tsx` + CSS): ⌘⌥D toggles a top-right panel that captures `click`/`pointerdown`/`pointerup`/`wheel`/`keydown` at document level with `capture:true` (so upstream `stopPropagation` can't hide events from us). 50-entry ring buffer, Clear/× buttons. Panel itself is `pointer-events:none` so it never steals what it's observing. Filtered to events whose target is inside `.bucket-workspace`.
- **Enter-to-focus model** (`BucketWorkspace.tsx`): removed the earlier auto-focus `createEffect` (it was racing with clicks and stealing focus on every ⌘⌥-navigate). Replaced with a window-level `keydown` listener that: on **Enter** with no xterm focused → focuses the facing terminal OR emits `notes-open` if the notes face is forward; on **Escape** with xterm focused → blurs the active element. Skipped when modifiers held (so ⌘⌥3 etc. flow through). Detection: `active.closest(".bw-pane.is-3d.is-facing .xterm")`.
- **Blur-on-navigate** (`BucketWorkspace.tsx`): a tiny `createEffect` that reacts to `activeTabId()` and `activeProjectIdx()` in 3D mode and `.blur()`s the active element if it's inside a `.bw-pane .xterm`. Closes the leak where Enter on the notes face would open the modal AND write a newline to the previously-focused terminal.
- **WebGL → Canvas fallback** (`TerminalPane.tsx`, `package.json`): added `@xterm/addon-canvas@^0.7.0`. Replaced `webgl.onContextLoss(() => webgl.dispose())` with a handler that disposes the dead addon AND loads `CanvasAddon` so the terminal keeps rendering. Initial WebGL-load failure also falls back to canvas. Surfaces gracefully when the page exceeds WebKit's ~16-WebGL-context cap (bucket workspace + multiple per-project windows together).
- **Window-scoped notes-open emit** (`src/lib/ipc.ts`): `emitMenuEventLocal` now passes the explicit `{kind:"WebviewWindow", label:w.label}` target object to `webview.emitTo`. The bare-string form silently builds `{kind:"AnyLabel"}`, and Tauri's runtime treats `AnyLabel` and `WebviewWindow` as distinct EventTarget kinds — listener filter dropped every event, so notes never opened. Captured the lesson in `~/.claude/projects/.../memory/feedback_tauri_event_target_kinds.md` so future-me doesn't repeat it.

Commit `615ae87` — "fix: NotesModal — Esc closes from any focus, via window listener":
- Existing `onKeyDown` was attached to `.notes-overlay` div and only fired when the event bubbled THROUGH the overlay. When the modal opens via Enter-to-focus, focus is on `document.body` (xterm was blurred, overlay never receives programmatic focus), so Esc never reached the handler. Switched to a window-level `keydown` listener inside a `createEffect` gated on `open()`. Overlay's `onKeyDown` kept as a redundant fallback. Also handles uppercase N/S for ⌘N/⌘S.

## What Was Tried and Abandoned
- **Approach**: `emitTo(stringLabel, ...)` to scope the notes-open emit to the current window.
  - **Why we walked back**: shipped, then NOTHING opened in any window. Diagnostic Explore agent dug into `node_modules/@tauri-apps/api/event.js` and confirmed `emitTo(string, ...)` builds `{kind:"AnyLabel"}` while `webview.listen()` filters for `{kind:"WebviewWindow"}` — the kinds don't cross-match. Fixed by passing the explicit object form.
  - **Lesson**: Saved to memory. Future event-system changes in this codebase MUST verify the listener side's target-kind matches the emit side's. Type-check alone won't catch it.
- **Approach**: Double-click `.pnr-item` to open (single-click previews, double-click opens).
  - **Why we walked back**: User wanted single-click-opens, simpler. Dropped before shipping; replaced with single-click handler that does both select-for-preview AND emit-open.
- **Approach**: Project chip at top-right.
  - **Why we walked back**: Overlapped with Re-scan/Debug/Enter-3D buttons in the auto-hidden header when revealed on hover. Moved to top-left, added fade-on-header-hover via sibling selector.
- **Approach**: Auto-focus on ⌘⌥-navigate (first version of focus handling).
  - **Why we walked back**: User explicitly requested the Enter-to-focus model — "as I navigate, don't take the focus immediately... press enter to go into the active viewing terminal". Auto-focus was also racing with click events on the notes pane and possibly with WebGL-lost canvases. Deleted in favor of explicit Enter / Esc gestures plus a reactive blur-on-navigate.

## Decisions Made
- **Enter-to-focus over auto-focus in 3D bucket workspace** — no ADR (small scope, only affects 3D mode in this one component). Captured in this handoff + STATUS recent-decisions. Per-project window's `TerminalsView` keeps its auto-focus behavior (single-window, no WebGL-cap pressure).
- **Window-scoped menu emits via explicit `EventTarget` object** — no ADR. Behavioral fix only. Lesson saved in `feedback_tauri_event_target_kinds.md` memory.
- **WebGL → Canvas fallback on context loss, not lazy-mount** — graceful degradation now; the deeper architectural fix (only mount facing pane's xterm) deferred to a future slice if canvas fallback proves insufficient.
