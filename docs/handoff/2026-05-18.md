# Handoff — 2026-05-18 [05:00 session start]

## Session Goal
User reported: clicking "Resume session" on a suspended Claude tab "most of the time" produced two duplicate terminal tabs both running `claude --resume` against the SAME UUID. Goal was to diagnose how two tabs were converging on a single session and ship the fix end-to-end (code → build → install).

## What Got Done
Commit `3380f1f` — "fix: per-tab Claude UUID binding to stop suspend/resume convergence":

- **New `claudeUuidByTab` signal** (`src/components/TerminalsView.tsx`): per-tab Claude session UUID, seeded at restore from each `listPersistedTerminals` row's `claude_session_id`, refreshed on every spawn via the post-spawn poller, and cleaned up in `closeTerminal`.
- **Post-spawn jsonl-diff polling** in `handlePaneSpawned`: snapshots the cwd's existing `.jsonl` set BEFORE the 250 ms post-spawn write, then polls at 1.5 s, +2 s, +2.5 s, +4 s (≈ 10 s total) for a UUID present in the cwd but NOT in the snapshot. That UUID is the NEW session `claude` created on resume, and gets bound to this specific tab. Aborts if tab closed or re-suspended mid-poll.
- **Defensive guard against handlePaneSpawned double-fire**: early-return + `console.warn` if `sessionIdByTab[tabId]` is already set. Symptom in the bug report (image 5 — `claude --resume` echoed twice in the same PTY) can only happen if the post-spawn write path fires twice. Guard catches it; the warn lets us trace the cause if it recurs.
- **`suspendTab` reads bound UUID first**: only falls back to `detectClaudeSessionsForCwd(cwd)` when unbound, and the fallback excludes UUIDs already owned by other tabs in the same cwd so two unbound siblings can't grab the same `[0]`. Caches the resolved UUID back into `claudeUuidByTab` for the next cycle.
- **Restore-time dedupe by (cwd, uuid)** in `TerminalsView.onMount`: defensive heal for any stale duplicate rows left in `persisted_terminals` by older builds. Drops them silently with a `console.warn`.
- **`persist_project_terminals` signature change** (`src-tauri/src/commands.rs`): now takes `Vec<PersistTabInput { cwd, claude_session_id }>` instead of `Vec<String>` cwds. Rust uses the per-tab UUIDs directly (no FS rescan), dedupes by (cwd, uuid), and skips any tab whose UUID's `.jsonl` no longer exists via `session_restore::session_file_exists`. The old "scan the cwd, write every UUID we find" implicit-discovery path is gone — same-cwd tabs now persist as N distinct rows, not as M-deduped-by-cwd rows.
- **IPC wrapper updated** (`src/lib/ipc.ts`): new `PersistTabInput` type; `persistProjectTerminals(projectId, tabs)`.

## What Was Tried and Abandoned
- **Approach**: Initial guess (from the first round of code reading) that the duplicate-tab symptom was a Solid `<Show>` rendering both branches transiently, or a missed `addTerminal` call somewhere in the resume path.
  - **Why we walked back**: User dropped screenshots showing two tabs both running `claude --resume f2b26e3f-…` with the SAME UUID. That ruled out "second tab created on resume" — both tabs were genuine `allTabs` entries with their own PTYs and post-spawn writes. The convergence was on the UUID, not on the tab count primary. Pivoted from "find the extra setAllTabs call" to "find why two tabs would ever resolve to the same UUID."
  - **Lesson**: Trust user screenshots as the ground truth. When the diagnosis says "this is impossible per the code" and the evidence shows it's happening, the impossibility lives in an unexamined assumption. Here it was "each tab has its own UUID" — actually the suspend path threw that away on every cycle by re-deriving from the cwd scan.
- **Approach**: Considered adding a long-running watcher (every 30 s) for late-typed manual `claude` so binding catches it.
  - **Why we walked back**: Adds polling cost across every tab forever; the user's actual broken case is auto-restore + auto-suspend, which the 10 s post-spawn window covers. Suspend's exclude-list fallback handles late-typed claude correctly within a session. Persist would miss it on close, but that's a smaller class of pain than the original convergence bug.
  - **Lesson**: Ship the minimum fix that handles the reported symptom + the broad class. Reserve the watcher for if the late-typed case actually bites someone.
- **Approach**: Considered making the persist Rust-side fall back to FS detect for tabs with no bound UUID.
  - **Why we walked back**: Re-introduces the cwd-scan that caused convergence. If we want a fallback, it has to be frontend-side with the exclude-list (same logic as suspend's fallback), not Rust-side blind detect. Punted to "if the late-typed case bites, do it then with the right structure."
  - **Lesson**: Don't restore a removed bug pattern in the name of completeness. The whole point of the per-tab signature change was to make Rust trust what the frontend tells it.

## Decisions Made
- **Bind UUID to tab, not to cwd.** Architectural shift — every code path that used to ask "what UUID is in this cwd?" now asks "what UUID has this tab bound?" Captured in STATUS recent-decisions; not promoted to a full ADR because it's a bug fix that aligns with the original intent (tabs were always meant to own their session, the cwd-scan was a shortcut that turned into a bug).
- **Drop implicit "scan cwd at persist, discover orphan sessions" behavior.** Tabs now only persist what they explicitly own. `load_active_claude_sessions_for_bucket` remains the explicit recovery path for orphan-session discovery. Captured in STATUS.
